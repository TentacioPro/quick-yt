# 01-monorepo-scaffold — Decisions

## Decision Log

### D-01 · @testing-library/react-hooks dropped in favour of direct Zustand `getState()`

**Context:** `@testing-library/react-hooks@8.0.1` declares peer deps on React `^16.9.0 || ^17.0.0`. The workspace uses React 19.2.3 (shipped by Expo 57), making it incompatible without `--legacy-peer-deps`.

**Decision:** Test `useToastStore` by calling `useToastStore.getState()` and `useToastStore.setState()` directly — no React renderer required. This is the canonical Zustand testing pattern and produces cleaner, faster tests with zero DOM overhead.

**Rejected path:** `--legacy-peer-deps` install + `renderHook` wrapper. Rejected because it introduces silent version drift and masks real compatibility issues.

---

### D-02 · `babel.config.js` added — not generated by blank-typescript template

**Context:** `create-expo-app --template blank-typescript` does not generate a `babel.config.js` in Expo 57. When Jest (via `jest-expo` preset) processes `@react-native/jest-preset/jest/setup.js`, it encounters an `import` statement that Babel hasn't been configured to transform.

**Decision:** Manually create `babel.config.js` with `babel-preset-expo`. This is the standard Expo Jest setup documented in expo.fyi/jest.

**Rejected path:** Switching to `ts-jest` exclusively. Rejected because `jest-expo` preset provides critical transform rules for RN modules and asset handling that `ts-jest` alone cannot replicate cleanly.

---

### D-03 · `transformIgnorePatterns` overrides the preset's default

**Context:** `jest-expo` preset emits `transformIgnorePatterns` tuned for its own registry. `zustand@5`, `drizzle-orm@0.45`, and `markdown-it@14` ship as ESM modules. Without explicit whitelist, Jest tries to run them as CommonJS and fails.

**Decision:** Override `transformIgnorePatterns` in `jest.config.js` using the verbatim base from the jest-expo preset output and appending `zustand|drizzle-orm|markdown-it` to the non-ignore list. The pattern is locked to avoid silent regressions on package upgrades.

---

### D-04 · `react-native-vector-icons` deprecated warning — acceptable

**Context:** `pnpm` reports `WARN deprecated react-native-vector-icons@10.3.0`. This package is a peer dependency of `react-native-paper`.

**Decision:** Accept as-is. `react-native-paper` is pinned to `^5.15.3` which requires this package. Migration to `@expo/vector-icons` is tracked but deferred. The deprecation is a metadata label only — the package is functionally sound for Expo 57.

---

### D-05 · `PaperProvider` + `SafeAreaProvider` both mounted at App root

**Context:** `react-native-paper`'s `<Snackbar>` requires being inside a `<PaperProvider>`. Snackbar positioning also depends on `SafeAreaProvider` for notch-aware rendering on Pixel 8a.

**Decision:** Wrap the entire app tree with `<SafeAreaProvider><PaperProvider>`. `GlobalSnackbar` is mounted as a sibling to the screen content (inside `PaperProvider` but outside screen views) so it overlays correctly.

---

### D-06 · Sync-server `pnpm-lock.yaml` note for Docker

**Context:** The Dockerfile copies `pnpm-lock.yaml*` (wildcard). In a monorepo, the lockfile lives at the workspace root, not inside `tools/sync-server/`. The Docker build context must be set to the workspace root.

**Decision:** Document this in a `docker-compose.yml` in Task 05. For Task 01, the Dockerfile is a valid placeholder. Full container build is validated in Task 05.

---
